# output-polish.md

## 1. 出力アーキテクチャの4層構造化

### Layer 1: Internal Structured Data
役割:
- 正史となる事実データ
- source ごとの scope、timeline、groups、candidate、decision 状態などを保持
- すべての出力の唯一の元データ

要件:
- canonical timeline を持つ
- 事実と推測を分ける
- source 名を正規化する
- 出力用 copy はまだ持たない

### Layer 2: Chat-side Output Contract
役割:
- その時点の生成結果、保存状況、不足情報、継続可能な操作を返す対話面
- ユーザーとのその場のやり取りを成立させる
- 曖昧性が残る場合は、必要に応じて確認質問を返す

要件:
- 最終成果物そのものを無理にここへ載せない
- 長すぎず、切れても致命傷にならない
- 「何が生成されたか」「どこまで保存できたか」「何が未確定か」を明示する
- 意思決定や継続操作に必要な最小限の情報に集中する
- mixed-scope 注記と再構成元の要約は **chat に必須**（§5-5 参照）。artifact 本文への挿入は必須としない
- ただし proposal の候補一覧は例外: 選択判断に必要な最低限の情報（番号 + ラベル + 種類 + 確度 + 効果の一言 + アクション導線）は chat に出す（§6 参照）

### Layer 3: Persisted Intermediate Artifacts
役割:
- セッション中および後続処理で利用される一時保存物
- その時点の生成結果をファイルとして保持する
- 再生成、追加生成、派生生成の足場になる

要件:
- Markdown を中心に保存する
- 保存先は §7 で定義する
- ファイル単位で上書き可能にする
- 1 つの artifact を作成した後でも、同一セッション内で他の artifact を継続生成できる
- この層は最終配布物そのものとは限らない

### Layer 4: Optional Downstream Exports（plugin 責務外）
役割:
- artifact をもとに外部向け・運用向けの形式へ派生出力する
- Google ドキュメント、テンプレ差し込み、スプレッドシート 1 行追加などの後段利用に備える

スコープ:
- Layer 4 は DayTrace plugin の責務外とする
- DayTrace は Layer 3 で安定した artifact shape を提供することに責任を持つ
- export は Claude Code エコシステム側のスキル（/docs, GAS 等）やユーザーの任意手段で行う
- plugin が export 先の制御やフォーマット変換を内包しない

連携可能な外部スキルの例:
- Claude Code /docs — Markdown ドキュメントの整形・配置
- GAS skill — Google ドキュメント・スプレッドシートへの差し込み
- その他ユーザー定義のスキルやスクリプト

連携の前提:
- Layer 3 artifact の shape（ファイル名・保存先・Markdown 構造）が安定していること
- export 側が artifact を消費する形をとること（DayTrace が export 先に push しない）

## 2. 設計の前提

この文書は、DayTrace の出力において次の混線を防ぐことを目的とする。

- 内部構造化データと、チャット上の説明と、ファイル保存物の責務が混ざること
- chat に最終成果物本文を過剰に載せて、長文化・切断・英語混入・要約崩れを誘発すること
- 一時保存 artifact と、外部共有や運用連携向け export を同一視すること
- 曖昧性解消の質問が、必要性ではなく慣習的な確認に流れること

plugin の責務境界:
- DayTrace の責務は「ローカル証跡の収集 → 構造化 → artifact 生成 → エージェント自己改善」で閉じる
- Source は Source CLI Contract に従えば追加可能（拡張に開いている）
- Export は Layer 3 artifact の shape が安定していれば外部スキルから消費可能（連携に開いている）
- ただし plugin 自体は export 先の制御を持たない

観測スコープと適用スコープの分離:
- skill-miner では「どこから観測したか」と「どこへ定着させるか」を別の軸として扱う
- `workspace` / `cwd` / `--all-sessions` は観測スコープであり、提案の適用先を自動的に決めるものではない
- `CLAUDE.md` / `skill` / `hook` / `agent` は適用手段の分類であり、適用スコープそのものではない
- `--all-sessions` は「広く観測する」ためのモードであり、「~/.claude のような全体設定だけを提案する」モードではない
- したがって、広域観測した結果からでも repo ローカル提案と個人グローバル提案は区別して扱う必要がある
- `--all-sessions` を付けても、**CWD / `--workspace` に紐づく source**（`git-history`、`workspace-file-activity` 等）は **引き続きその workspace 視点**の観測である。横断モードでも **repo 固有の兆候と個人横断の兆候が同居しうる**ため、`workspace-local` と `global-personal` で混同しない
- chat の proposal **compact 表**では、`| 種類 |`（`CLAUDE.md` / `skill` / `hook` / `agent`）に加え、**`| 適用スコープ |` 列を載せてよい**（任意。最小限の例示でよい。§9-4）

### 2-1. 日付指定と観測スコープ（日報・投稿下書き・aggregate）

- **日付は指定できる**: CLI / SKILL では `YYYY-MM-DD`、`today`、`yesterday` 等で対象日を渡せる（§7 の報告日ルールと `resolve_date_filters` が一致する）
- **未指定時のデフォルト**: `today` 相当。報告日は §7（06:00 境界）に従う
- **workspace の既定**: aggregate / projection は **カレントディレクトリ（cwd）を workspace として扱う**のが基本。必要なら `--workspace /absolute/path` で明示する
- **`--all-sessions`**: Claude / Codex 等の **all-day 系 source で「セッション横断の観測」**を広げるスイッチ（Source CLI Contract および各 SKILL の Data Collection に従う）。**適用先を自動決定するフラグではない**（上記「観測スコープと適用スコープの分離」参照）

### 2-2. 適用パス（skill-applier）

proposal の `suggested_kind` に応じ、**`skill-applier` skill** が定型的な適用導線を持つ（詳細は `plugins/daytrace/skills/skill-applier/SKILL.md` と `references/`）。

| 種類 | ざっくりした流れ |
|------|------------------|
| `CLAUDE.md` | diff preview → ユーザー確認後に追記・整備（immediate apply path） |
| `skill` | scaffold / 構造化 context の提示 → **公式の skill-creator（handoff）** へ渡すパス（plugin 内で skill ファイルを勝手にデプロイしない） |
| `hook` | 設計案 → 承認後に settings 等へ guided creation |
| `agent` | 設計案 → 承認後に `agents/` 等へ guided creation |

daytrace-session の Phase 4（Pattern Mining）で候補選択があった場合も、この skill に委譲して閉じる。

### 2-3. OAuth・クラウド認証を source に含めない理由と、artifact で止める意味

- **プラグインに OAuth やクラウド API 認証を持ち込まない**: トークン保管・同意画面・失効処理が設計の中心に乗り、**「設定不要・ローカル完結」**という DayTrace の前提と衝突する
- **収集対象はローカルに既にある証跡に限定する**（Git、ローカル AI ログ、Chrome DB の読み取り専用コピー、ワークスペース上のファイル痕跡など）。これにより **ネットワーク上の「生のアカウントデータ」を引きずり込まない**
- **Layer 3 の Markdown artifact で止める**: SaaS への自動 push やテンプレ埋め込み用 JSON の生成は **Layer 4（export）**やユーザー環境の別 skill に任せ、DayTrace は **読めるファイルと安定した shape** を出すことに専念する。認証が要る先への書き込みは、**ユーザーが明示的に別ツールを使う**前提に分離する

設計方針:
- 出力は 4 層に分けて考える（Layer 4 は plugin 責務外）
- chat は対話面、artifact は一時保存面として分離する
- Ask は曖昧性が残る場合だけ、自律的に発火する
- 保存・失敗・継続可能性はファイル単位で伝える
- **DayTrace core（plugin）**は安定した **Markdown artifact** を出すことに集中する
- **export 側（Layer 4）**が、出力先に合わせて **JSON / placeholder マップ / 行データ** などを生成する（plugin は内包しない）

## 3. 解決したい問題

現状の出力で起こりやすい問題は次の通り。

- チャット応答に成果物本文を寄せすぎて長くなり、読みにくくなる
- chat 側の短縮や圧縮で情報が切れる
- 日報・投稿下書き・提案候補の役割が混ざる
- 「何が生成済みか」「どこまで保存できたか」が曖昧になる
- 曖昧性解消のための質問がなく、出力の質が落ちる
- 逆に、聞かなくても出せるものまで質問し、エージェントの自律性を損なう
- 一時保存 artifact をそのまま最終成果物と誤認しやすい

## 4. 出力物ごとの役割分担

### 4-1. Internal Structured Data
内部では、全 source から再構成された事実データを canonical timeline に束ねる。
この層は出力本文を持たず、事実・推測・候補・判断状態などの材料を保持する。

### 4-2. Chat-side Output
chat はその場での対話を成立させるための出力であり、次の情報を扱う。

- 何が生成されたか
- どのファイルに保存できたか
- 何が未確定か
- 曖昧性解消のために何を聞くか
- 続けてどの出力を生成できるか

chat は成果物本文の代替ではない。

### 4-3. Persisted Intermediate Artifacts
artifact はセッション中・後続処理中の足場となる一時保存物である。

例:
- `report-private.md`
- `report-share.md`
- `post-draft.md`
- `proposal.md`

これらは後続の再生成や追加生成に利用できる。

### 4-4. Optional Downstream Exports（plugin 責務外）
artifact をもとに、必要に応じて外部向けや運用向けの形式へ派生出力する。
この層は DayTrace plugin の責務外であり、外部スキルやユーザーの任意手段で行う。

連携例:
- Claude Code /docs による Markdown 整形・配置
- GAS skill による Google ドキュメント・スプレッドシートへの差し込み
- 共有用 Markdown / HTML への変換
- export 側が artifact から **structured fill 用 JSON**、**プレースホルダマップ**、**スプレッドシート行** などを生成する

### 4-5. セッション完了サマリ
目的:
- その時点で何が生成され、何が保存され、次に何ができるかを短く伝えること

理想:
- 日本語で統一
- 内部推論や英語混入なし
- 生成結果と保存状況を混同しない
- ユーザーが今決めることと、DayTrace がすでに行ったことを分けて書く

## 5. 出力品質ガード

### 5-1. canonical timeline
すべての出力は canonical timeline を基準に組み立てる。
source ごとの断片をそのまま並べず、同一時系列上で整列させる。

要件:
- **並べ方は時系列（古い→新しい、またはイベント時刻順）を基本とする**
- 見出しや本文に **時間帯ラベル（午前/午後など）を必須とはしない**。付ける場合もざっくりでよい
- 時間帯スロットの境界は厳密に定めず、**ざっくりした定義でよい**（詳細・表形式は daily-report / post-draft の SKILL.md に委ねる）
- 共有用日報（report-share）も **時系列を基本**とし、カテゴリ内の項目順がログ上の時刻と明らかに矛盾しないようにする（カテゴリ構成は SKILL に従う）
- 同一イベントの重複記述を避ける
- source 別の見え方ではなく、ユーザー視点の時間軸でまとめる
- 事実と推測が混ざる場合は、推測であることを明示する

参考（ざっくりスロットの例。SKILL で別定義してよい）:

| スロット | 時間帯（目安） |
|---------|----------------|
| 深夜 | 00:00–06:00 |
| 午前 | 06:00–12:00 |
| 昼 | 12:00–14:00 |
| 午後 | 14:00–18:00 |
| 夜 | 18:00–24:00 |

上表は参考例であり、SKILL.md で境界やラベルを別定義した場合は **SKILL を優先**する。

### 5-2. 事実 / 推測 / 禁止語
出力上の文章では、次を区別する。

- 事実: source から裏づけ可能なこと
- 推測: source から直接断定できず、再構成や補完を含むもの
- 禁止語: 内部状態や内部トレースを露出する語

例:
- 出してよい: 「この時間帯は issue 調査に多くの時間を使っている」
- 推測明示が必要: 「主作業はリサーチだった可能性が高い」
- 出さない: `candidate_id`, `triage_status`, `trace`, `internal state`

プロダクトコピー上の禁止・置換（例: 「寄り道」「今日の重心」など）は、上記の「内部状態」禁止とは別枠とし、**各 SKILL.md の語彙ルールに委ねる**（本書では列挙を必須としない）。

### 5-3. internal English leakage
出力本文への英語混入は避ける。
特に chat や日報本文には、内部設計語や処理系の中間語をそのまま出さない。

許容:
- 固有名詞
- コマンド名
- ファイル名
- 仕様上必要なラベル

非推奨:
- 内部処理語を英語のまま本文に混ぜること
- reasoning 由来の中間文をそのまま流すこと

### 5-4. 低信頼度データの扱い
source によって信頼度が異なるため、出力での扱いを区別する。

高信頼度（主軸として使用可）:
- Git ログ（commit, diff, branch）
- AI 会話ログ（Claude, Codex 等）

低信頼度（補助・裏づけ限定）:
- Chrome 閲覧履歴
- workspace-file-activity（ファイル変更検知）

ルール:
- 低信頼度 source の情報は、高信頼度 source を補強する文脈でのみ使用する
- 低信頼度 source 単独で主張の根拠にしない
- 低信頼度 source しか存在しない時間帯は「記録が限定的」と明示する
- 同一事実に複数 source がある場合は、高信頼度側を優先して記述する

見せ方の制御:
- 本文の主張では、高信頼度 source を優先して簡潔に記述する
- 根拠欄や注記では、低信頼度 source の存在を可視化してよい
- ただし低信頼度 source を見せる場合は、「補助推定」「閲覧ログベース」「記録が限定的」などのラベルを必ず付ける
- 低信頼度 source の raw 羅列は避け、読める粒度に短く要約する
- 「信頼感のために見せる」ことと「主張の根拠にする」ことを分離する

表示レイヤごとの原則:
- chat 要約: low データは必要な時だけ短い注記で見せる
- artifact 本文: low データは本文の主役にしない
- artifact の根拠欄: low データの存在を明示してよい
- 構造化データ: source ごとの信頼度差を保持したまま残す

### 5-5. Mixed-scope 注記と再構成元（chat 必須 / artifact 非必須）

信頼性と誤解防止のため、**DayTrace セッション完了時の chat** には、次の 2 点を**必ず**含める（各 1–3 行・箇条書きでよい。長いフッター全文は不要）。

1. **Mixed-scope 注記**: 1 日のログと workspace ローカルの変更ログなど、**複数種のソースを横断して再構成している**旨
2. **再構成元の要約**: 参照したソースの種類（例: Git / AI 会話ログ / 閲覧履歴 等）を人が読める短い表現で示す

**Markdown artifact（report-private / report-share / post-draft 等）の本文には、上記 2 点の挿入を必須としない**（読みやすさ優先。SKILL で任意に入れてよい）。

## 6. Chat-side Output Policy

chat は最終成果物そのものではなく、その時点の生成結果・保存状況・不足情報・継続可能な操作を返す対話面とする。

原則:
- 完全な成果物本文をそのまま流さない
- 生成物の概要だけを伝える
- 意思決定や継続操作に必要な部分だけを見せる
- 曖昧性が残る場合だけ確認質問を出す
- persisted artifact がある場合は、chat はその存在と状態を明示する

chat に出すべきもの:
- source 収集結果の要約
- DayTrace ダイジェスト
- **mixed-scope 注記と再構成元の要約**（§5-5。セッション完了系のまとめの中で必須）
- 各 artifact の生成結果
- 各 artifact の保存状況
- 曖昧性が残る場合の短い確認質問
- proposal の候補一覧（選択判断に必要な compact 表。§2 のとおり **`適用スコープ` 列は任意で載せてよい**）
- セッション継続時に次に出せる出力の案内

chat に出さなくてよいもの:
- 完全な Markdown 成果物本文
- 長い根拠一覧
- フッター全文
- 内部 trace
- 内部状態語（candidate_id, triage_status 等）

chat に出してはいけないもの:
- `[DayTrace] Phase N: ...` 等のオーケストレーションメタデータ
- 内部スクリプト名（skill_miner_prepare.py 等）をそのまま露出する記述
- 処理フェーズの進行状況を示す内部ログ形式の文字列

proposal の compact 表の形式（`適用スコープ` 列は任意）:

```text
| # | 候補 | 種類 | 適用スコープ（任意） | 確度 | 効果 | アクション |
|---|------|------|----------------------|------|------|-----------|
| 1 | DayTrace 出力 UX ルール | CLAUDE.md | workspace-local | 高い | 次回の出力揺れを減らす | すぐに追加可 |
| 2 | 評価プロンプト改善 | skill | global-personal | 高い | 提案の比較精度が上がる | /skill-creator で生成可 |
```

### skill-miner の選択 UI

`提案（アクション候補）` が 1 件以上ある場合、候補選択は次の優先順位で提示する。

1. 選択式 UI（choices）
2. 候補ラベル入力
3. 番号入力

方針:
- 選択式 UI が使える実行環境では、番号打鍵ではなく choices による選択を優先する
- choices の表示ラベルは `候補名 + 種類 + 確度` が一目で分かる形にする
- choices が使えない環境では、候補ラベルまたは番号入力にフォールバックする
- chat には compact 表を出して判断材料を揃えつつ、実際の選択操作は UI 側で受ける
- この選択 UI は skill-miner 単体の責務ではなく、daytrace-session 側の対話オーケストレーションで扱う

選択肢ラベル例:
- `DayTrace 出力 UX ルール / CLAUDE.md / 高い`
- `評価プロンプト改善 / skill / 高い`

chat の見出しは、散文的な要約見出しよりも状態を表す見出しを優先する。

例:
- 生成結果
- 保存状況
- 確認が必要な点
- フォールバック方針
- 続けて出力できるもの

### chat 出力順序

chat に artifact を提示する順序を以下に固定する。
実行順序（Phase）とは独立に、ユーザーが読む順序として最適化する。

1. source 収集結果の要約
2. DayTrace ダイジェスト（3–5 行の散文概要）
3. 日報（自分用 → 共有用の順。artifact 保存状況を付記）
4. 投稿下書き（artifact 保存状況を付記）
5. skill-miner 提案（compact 表 + 有望候補 + 観測ノート）
6. セッション完了サマリ

理由:
- 日報と投稿下書きは「今日何をしたか」の出力であり、連続して読むのが自然
- skill-miner は「パターン検出・自己改善提案」であり、日報/下書きとは関心軸が異なる
- skill-miner を最後に置くことで、「今日の記録 → 共有可能な物語 → 次への改善」という流れになる
- daytrace-session の実行順は Phase 3: post-draft → Phase 4: skill-miner とし、chat 提示順と一致させる（旧: skill-miner が先で読み順が切れた）

## 7. Artifact Storage Policy

persisted intermediate artifact の保存先と命名規約を固定する。

保存先:
- `~/.daytrace/output/YYYY-MM-DD/`

**報告日（ディレクトリ名の YYYY-MM-DD）の決め方:**

- 指すのは「暦上の今日」ではなく、**この実行でまとめる対象とみなす日**（いわゆる日報の日付）
- **デフォルト（ユーザーが日付を明示指定しない場合）**: ローカル時刻が **06:00 未満**なら **前日の暦日**、**06:00 以上**なら **当日の暦日**とする（深夜に日付が変わった直後のセッションが「まだ前日の日報」になるようにする）
- **ユーザーが日付を明示した場合**（引数・プロンプト等）はそれを優先する（将来の CLI / スキル拡張用）

命名規約:
- `report-private.md` — 自分用日報
- `report-share.md` — 共有用日報
- `post-draft.md` — 投稿下書き
- `proposal.md` — 提案の詳細版（候補ごとのフル説明）

ルール:
- ディレクトリが存在しない場合は作成する
- 上書きはファイル単位で行う
- 同日に複数回実行した場合は、対象ファイルのみ上書きする
- 一部のファイルだけ更新されてもよい
- 既存 artifact を足場に、同一セッション内で追加生成を継続できる
- chat 出力時に、対象ファイルごとの保存状況を明示する

**同梱ファイル（plugin 責務内で書いてよいもの）:**

- **`aggregate.json`（任意）**: 内部デバッグ、再生成・調査のための source snapshot として **残してよい**。運用テンプレ差し込み用の主データとはみなさない
- **structured fill 用 JSON**: plugin では **出力しない**。Layer 4 の export 側が Markdown artifact を読み、**出力先に合わせて** JSON / プレースホルダマップ / 行データを生成する

artifact format の合意:
- artifact は人間がそのまま読める Markdown を基本とする
- 同時に、外部スキルや後続エージェントが消費しやすい stable shape を保つ
- 「人間可読」と「export しやすさ」を両立するため、見出し構造・保存先・ファイル名を固定する
- テンプレ埋め込み用の構造化データは **export 側の責務**とし、必要なら Markdown をパースして別成果物にする

## 8. Failure / Degrade Policy

出力失敗時の挙動を定義する。

### 8-1. テキスト切れへの対処
- 根本対策は、artifact をファイルに書き出すことで chat のトークン上限や圧縮の影響を回避すること
- chat には要約のみを出すため、chat 側でテキスト切れが起きても artifact 側は無傷であることを優先する
- ファイル書き出し自体が失敗した場合は、chat に短い要約と対象ファイルごとの失敗を返す

### 8-2. 保存失敗時
- 成功したファイルだけ保存する
- 失敗はファイル単位で明示する
- 「一部保存しました」のような曖昧表現は避ける

例:
- report-private.md を上書き保存しました
- post-draft.md は保存に失敗しました
- proposal.md は今回未更新です

### 8-3. 曖昧性が残る場合
- 出力品質または正確性が有意に落ちると判断した場合のみ Ask を発火する
- Ask が解消しなくてもフォールバックで継続する

### 8-4. その他の失敗
- internal English leakage は §5-3 の出力品質ガードで防止する
- 部分成功時は、成功分を成果として残し、未完了分だけを明示する

## 9. 出力種別ごとの指針

### 9-1. report-private
- 自分用の日報
- 事実ベースを優先しつつ、必要なら軽い再構成を許可する
- canonical timeline に沿って整理する
- 内部用なので多少の詳細を保持してよいが、内部状態語は避ける

見出し構造:
```
## 日報 YYYY-MM-DD
### 今日の流れ
  1. 見出し / 内容 / 根拠:
### 未完了の手がかり
```

### 9-2. report-share
- 共有用の日報
- 内輪情報や不要な詳細を落とす
- 公開・共有して問題ない表現に寄せる
- 「何をしたか」「なぜ重要か」が伝わる密度に整える

見出し構造:
```
## 日報 YYYY-MM-DD（共有用）
### 今日の概要
### 実装
### 調査（任意）
### 設計 / 判断（任意）
### 未完了の手がかり
### 参考: 根拠一覧
```

注意:
- 見出しは SKILL.md 仕様に従い「今日の概要」を使う（「今日の活動」は不可）
- カテゴリ分割（実装 / 調査 / 設計・判断）を省略しない
- 根拠は本文から分離し「参考: 根拠一覧」にまとめる

### 9-3. post-draft
- 投稿下書き
- ログや出来事から、公開可能な範囲で自然な主題を立てる
- 方向性ラベルや固定選択肢をユーザーに要求しない
- 足りない情報があり、曖昧なままでは質や正確性が落ちる場合だけ Ask する

見出し構造（標準）:
```
# タイトル案（8–20 字）
## 背景
## 今日の中心
## 気づき（根拠が弱い場合は省略可）
```

見出し構造（非技術者向け）:
```
# タイトル案
## 何に取り組んだか
## 今日の中心
## 気づき
```

見出し構造（調査中心の場合）:
```
# タイトル案
## 動機
## 比較したもの
## 判断
## 結論
```

### 9-4. skill-miner 提案
- skill-miner の chat 出力は compact 表（§6 参照）を基本とする
- 詳細版は `proposal.md` artifact に保存する
- 観測スコープと適用スコープを混同しない

適用スコープ（候補の分類）:
- `workspace-local` — 現在の repo / cwd に閉じた作法・ルール・手順
- `global-personal` — 個人の横断的な作法・原則（例: `~/.claude` 相当）

`global-personal` として提案しうる**種類**（手段の分類）は、`CLAUDE.md` / `skill` / `hook` / `agent` を含めてよい。`workspace-local` 同様、観測スコープ（`--all-sessions` 等）とは独立に判定する。

判定原則:
- repo 固有の制約、ファイル構成、運用ルールに強く依存する候補は `workspace-local`
- 複数 workspace で繰り返し観測され、repo 固有事情に依存しない候補は `global-personal`
- `--all-sessions` は `global-personal` を観測しやすくするが、`workspace-local` 候補を禁止しない
- workspace モードでもグローバル候補の兆候が見えることはあるが、確証が弱い場合は保留または「有望候補」に留める
- `CLAUDE.md` 提案は現状 repo ローカル前提で扱いやすいが、**個人グローバル向けの同種提案**は `global-personal` として区別してよい（混同しないこと）

chat / compact 表での表示:
- `適用スコープ` 列は**任意**。付ける場合は **最小限の例示**でよい（必須列にしない）
- 例: `CLAUDE.md / workspace-local`、`rule-set / global-personal`

見出し構造:
```
### 観測範囲
## 提案（アクション候補）
## 有望候補（もう少し観測が必要）
## 観測ノート
```

### 9-5. 複数選択

現時点では仕様を未確定事項として扱う。

確認したい点:
- 同一セッションで複数 artifact を連続生成する操作を、1 レスポンス内の複数選択として扱うのか
- それとも 1 回 1 操作を積み重ねる対話として扱うのか
- 既存 artifact 群との依存関係や上書き順序をどう扱うのか

daytrace-session との関係:
- daytrace-session は Phase 1–5 でスキルをオーケストレーションする
- 複数 artifact の生成順序は daytrace-session の Phase 制御に従う
- ただし Phase メタデータ（`[DayTrace] Phase N: ...`）はユーザー出力に含めない（§6 参照）

当面の方針:
- まずは 1 回 1 操作の連続対話で破綻しないことを優先する
- 複数選択対応は continuation model が固まってから導入する

## 10. Ambiguity Resolution Policy

自律性を損なわずに品質を上げるため、曖昧性解消の扱いを定義する。

方針:
- Ask は、出力品質または正確性に影響する曖昧性が残っている場合にのみ発火する
- Ask はエージェントが自律的に必要性を判断する
- Ask は **出力種別ごとに最大 2 ターンまで**（例: 日報と投稿下書きでそれぞれ最大 2 ターンまで。セッション全体の厳密な合算上限は設けない）
- どの出力種別でも発火しうる
- 無回答または解消失敗時はフォールバックで続行する

原則:
- 「聞かなくても出せるなら聞かない」
- 「曖昧なまま進めると質または正しさが落ちる場合だけ聞く」
- Ask は短く、何を解消したいのかを明示する
- ユーザーに構成や文体の責任を戻さない

Ask の対象になりうる例:
- どの出来事を主軸にするか不明
- 公開してよい範囲が不明
- 内輪情報を伏せる必要があるか不明
- 同名候補や複数解釈があり、誤適用の恐れがある

対象にしないもの:
- 単なる好みの差
- 聞かなくても十分な品質で自動補完できるもの
- 技術寄り / 振り返り寄り / 発見寄り などの抽象的な方向性選択

## 11. 仕様矛盾の解消ログ

以下の矛盾を v2 内で解消済み。

| # | 矛盾 | 解消方法 |
|---|------|---------|
| M1 | Layer 2 が Semi-Final だと chat に本文を載せる設計へ戻りやすい | Layer 2 を Chat-side Output Contract に変更し、対話面として再定義 |
| M2 | Layer 3 を final artifact と呼ぶと、一時保存と最終配布物が混線する | Layer 3 を Persisted Intermediate Artifacts に変更し、export を Layer 4 として分離 |
| M3 | Optional Ask Policy が投稿下書き専用かつ方向性質問前提になっていた | Ambiguity Resolution Policy に変更し、全出力種別での自律質問に一般化 |
| M4 | 上書きが日単位なのかファイル単位なのか曖昧だった | §7 と §8 でファイル単位 overwrite / failure messaging を明文化 |
| M5 | 複数選択の話が UI 仕様に見えていた | continuation model の未確定事項として整理し、別論点に分離 |
| M6 | Layer 4 (export) を plugin 責務内に含めると責務が肥大化する | Layer 4 を plugin 責務外と明示し、artifact shape の安定性で外部連携を担保する設計に変更 |
| M7 | chat 出力順序が Phase 実行順に引きずられ、日報と投稿下書きの間に skill-miner が挟まる | chat 提示順を Phase 実行順から分離し、日報 → 投稿下書き → skill-miner の順に固定 |

## 12. 実装優先順

依存関係を考慮して再編した順序。
Layer 4 (export) は plugin 責務外のため、実装対象から除外する。

優先度高:
1. Artifact storage policy の実装（ディレクトリ作成 + ファイル単位 Write/Overwrite、§7 の報告日ルール含む）
2. 保存結果メッセージのファイル単位化
3. Chat-side Output Policy の実装（生成結果 / 保存状況 / 不足情報 / 継続操作 / **chat 出力順序** / §5-5 の chat 必須注記）
4. Canonical timeline ルールの追記（daily-report / post-draft SKILL.md）
5. 事実 / 推測 / 禁止語ルールの追記（各 SKILL.md）

優先度中:
6. 出力品質ガードの SKILL.md 追記（低信頼度データポリシー含む）
7. 見出し構造の SKILL.md 準拠徹底（§9 の見出し仕様を SKILL.md に反映）
8. 根拠 source 名の正規化
9. proposal 候補説明の差別化
10. 曖昧性解消 Ask の発火条件実装（出力種別ごと最大 2 ターン）
11. `aggregate.json` の扱い整理（デバッグ・再利用用の任意同梱に限定）

優先度低:
12. 複数選択対応
13. continuation model の明文化
